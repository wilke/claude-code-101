# 2026-05-11 — Baseline classifier for `outcome`

First honest pass at predicting the binary `outcome` from `experiment.csv`. Goal
was a defensible baseline, not a leaderboard number.

- **Decision (estimator).** `LogisticRegression(max_iter=1000)` as the baseline. It's
  interpretable, fast, and a linear model is the right first thing to beat before
  reaching for a random forest. RandomForest tried informally — roughly the same
  ROC-AUC, so no reason to abandon the simpler model yet.
- **Decision (features).** Dropped `outcome_label` and `sample_id` from `X`.
  `outcome_label` is the target restated as success/failure (see 05-12); `sample_id`
  is a row id, not a feature. Kept `treatment_group`, `batch` (one-hot),
  `temperature_C`, `concentration` (median-impute + scale).
- **Decision (evaluation).** Report **stratified 5-fold CV** with per-fold spread,
  not a single split — the dataset is small (~400 rows) so one split is noisy. Held
  out a stratified 20% test set for a final honesty check.
- **Decision (primary metric).** F1 is the primary metric this round because the
  target is imbalanced (~30% positive); accuracy is reported but not decisive. Also
  track ROC-AUC as a threshold-independent view.
- **Parameters.** `random_state=0` everywhere; `n_splits=5`; `test_size=0.20`;
  `LogisticRegression` default `C=1.0`.

**Numbers (leak-free baseline).** CV: accuracy ≈ 0.78, F1 ≈ 0.61, ROC-AUC ≈ 0.83.
Held-out test agreed (ROC-AUC ≈ 0.82) — the CV/test agreement is my evidence the
pipeline isn't leaking.

**Open question:** the positive class is only ~30%. Is F1 at the default 0.5
threshold the right target, or should we tune the threshold / use class weights?
Deferred — see 05-13.
