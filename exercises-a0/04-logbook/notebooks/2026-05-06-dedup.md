# 2026-05-06 — Duplicate rows

Found that `sample_id` is **not unique** — a few ids appear twice. Looked closer:
the repeats are *whole-row* duplicates (every column identical), not two different
measurements that happen to share an id. So this is a genuine duplication (an
export ran twice), not a data-model question.

- **Decision.** Drop full-row duplicates with `df.drop_duplicates()`, which keeps
  the first occurrence. This took 403 rows → 400. Report the number removed.
- **Why not dedupe on `sample_id` alone?** Because that would silently discard a
  row if two *different* measurements ever shared an id by accident — we'd lose
  real data. Dropping only exact full-row duplicates is the safe rule; if we later
  find same-id/different-values rows, that's a separate decision (which reading
  wins?), not a dedup.
- **Parameter.** Post-dedup row count: 400 unique rows.

**Open question:** should `sample_id` be enforced unique going forward (a
constraint at export time), so duplicates are impossible rather than cleaned up
after the fact? Cleaning the same problem every load is a smell.
