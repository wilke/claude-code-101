# Plan: FilterADMM walkthrough notebook + baseline comparison

## Context

`admm_walkthrough.ipynb` is the existing teaching notebook for the **baseline**
`admm.py` driver: 17 cells, six sections building from compliance вҶ’ ADMM math вҶ’
each subproblem вҶ’ full ADMM loop вҶ’ Оұ-sweep. `STATUS.md` "What's next" item 3
flags adding a `FilterADMM.py` walkthrough as the next documentation task.

This plan creates a parallel notebook **`filter_walkthrough.ipynb`** that:

1. Motivates the filter (baseline ADMM oscillates near convergence on the
   30Г—10 problem).
2. Walks through the new primitives `Filter`, `compute_eta`,
   `compute_omega`, `project_box_budget`, and the restoration phase.
3. Runs `filter_admm_compliance_tv` end-to-end with the same plotting style
   as the existing PDF outputs (`images/convergence.pdf`, `images/iter_NNNN.pdf`)
   but inline.
4. Ends with a head-to-head comparison: same problem, same `eps_opt`, both
   drivers вҖ” showing the baseline failing to converge where FilterADMM does.

Following the existing pattern, the notebook is generated by a Python script
(`generate_filter_notebook.py`, sibling to `generate_notebook.py`) вҖ” the
`.ipynb` file is never edited by hand.

### В§1 вҖ” Motivation: why a filter? (1 md + 1 code = 2 cells)

### В§2 вҖ” The augmented-Lagrangian filter (1 md + 1 code = 2 cells)

- **Md.** Acceptance condition `\eqref{eq:filter}`: a point $(О·,Пү)$ is
  accepted iff for every existing entry $(О·_l, Пү_l)$ either $О· вүӨ ОІВ·О·_l$ OR
  $Пү вүӨ Пү_l - ОіВ·О·$. Dominance pruning (an entry dominated on both axes is
  removed). Pointer to `RASC-MeetingNotes В§3.1`.
- **Code.** Demo: create `Filter(beta=0.99, gamma=1e-5)`, add 3 contrived
  entries, then plot the acceptance region in (О·, Пү) space:
  - Filter entries as black dots.
  - Rejected region shaded grey (union of "L"-shaped corners per entry).
  - 6 test points: green вң“ for accepted, red вң— for rejected, with
    `is_acceptable` calls printed underneath as a sanity check.

### В§3 вҖ” Optimality measures О· and ПүМғПҒ (1 md + 1 code = 2 cells)

- **Md.** Definitions of `О· = вҖ–x-yвҖ–ВІ` `\eqref{eq:eta}` and `ПүМғПҒ`
  `\eqref{eq:omega-prox}` вҖ” the proximal-gradient extension for the
  non-smooth `ОұВ·TV(y)` term. Brief explanation that the literal
  projected-gradient `Пү_ПҒ` is undefined for non-smooth y, so the y-block
  uses a prox-step instead. Mention the В§3.3 identity:
  $\widetilde{Пү}_ПҒ(x^{k+1},y^{k+1},О»^k) = \widetilde{Пү}_0(x^{k+1},y^{k+1},О»^{k+1})$
  after the multiplier update.
- **Code.** Along the baseline trajectory stored in В§1's `info_admm`:
  reconstruct (x, y, О») at each iteration from `info_admm` (the baseline
  driver stores `y`, `lam`; we recompute `x` from saved snapshots вҖ” or, if
  not available, re-run with a callback). Compute О· and ПүМғПҒ at each. Plot
  on log-y axis, alongside the primal/dual residuals from В§1. The point:
  О· and Пү don't converge вҖ” the residual-based stopping criterion gave up
  before the *real* first-order measures stabilised.
  *Implementation note:* `admm.admm_compliance_tv` doesn't currently
  store x-trajectories. Easiest: re-run with an `outer_callback` that
  appends `x.copy()` to a list, then compute О·/Пү post-hoc. (No edit to
  `admm.py` вҖ” the callback hook already exists per CLAUDE.md.)

### В§4 вҖ” Restoration phase (1 md + 1 code = 2 cells)

- **Md.** Three restoration triggers:
  1. eq.~(8): $О· вүҘ ОІВ·U$ (large primal infeasibility)
  2. eq.~(9): $Пү вүӨ Оө вҲ§ О· вүҘ ОІВ·О·_{\min}$ (dual converged but primal stale)
  3. $j+1 = M_{\max}$ (inner-iteration cap exhausted)
  Backtracking $\hat y = x + 2^{-m}(y - x)$ shrinks О· geometrically toward
  0; the first filter-acceptable $m$ is accepted, and ПҒ is doubled.
  Briefly enumerate the four departures + ПүМғвӮҖ correction with pointers to
  `RASC-MeetingNotes В§4.1вҖ“В§4.5`.
- **Code.** Contrived demo: take an (x, y) pair with `О· вүҲ 1` from the
  baseline run (e.g. iter 5 from В§1's info), pretend it's a candidate.
  Compute О· for the original (x, y) and for the backtracked $\hat y$ at
  $m=1,2,3,4,5$. Plot О· vs m on log-y axis showing the geometric decrease.
  Also show that the filter (with the entries from В§2) rejects the
  original but accepts $\hat y$ at some $m^* > 0$.

## Post-implementation notes (added 2026-05-18 after the run)

All verification steps from the plan passed. The implementation matched
the plan structure exactly (18 cells, six sections, single 30Г—10 mesh
throughout). Two issues caught during execution and fixed:

1. **`info["n_iter"]` vs `len(info["eta_history"])` off-by-one.** In
   FilterADMM, the convergence check at the top of the outer loop
   `break`s before the per-iteration history appends вҖ” so `n_iter` is
   one more than the history length. Plot cells using `np.arange(1,
   info["n_iter"]+1)` for the x-axis mismatched the y-axis array lengths
   and crashed in `ax.semilogy`. Fixed by using
   `np.arange(1, len(info["eta_history"]) + 1)`. The baseline `admm.py`
   does not have this off-by-one (its appends happen *before* the break
   check), so В§1 still uses `info_admm["n_iter"]` as written.

2. **`Filter.entries` is private.** Initial draft used `flt.entries` for
   the В§2 acceptance-region plot. The Filter class exposes entries via
   `__iter__` instead (`for e, w in flt:` or `list(flt)`); the private
   `_entries` attribute isn't part of the contract. Fixed by switching
   to iteration.

Final run on the 30Г—10 default (executed via
`jupyter nbconvert --execute`):

- Notebook: 18 cells, 11 code + 7 markdown, 7 inline PNG plots, no cell
  errors. 741 KB on disk.
- Baseline ADMM in В§1: 40 iterations (max_iter cap), did not converge.

- FilterADMM in В§5: 14 outer iterations, converged, 3 restorations

  post-`ПүМғвӮҖ`-correction numbers in `RASC-MeetingNotes.tex` В§4.6.
