Exercise 1 · CLAUDE.md

Write your first CLAUDE.md

Your first contact with file-based collaboration: encode your conventions for an FEM convergence study in one markdown file, then watch the assistant respect them the second time.

PDE supplement · v1

Exercise 1 · framing

Why an FEM author needs CLAUDE.md

Numerical PDE work is full of recurring micro-decisions — small, stable choices that shape every result. CLAUDE.md is where they live, so you don't re-justify them every prompt.

DecisionWhy it belongs in CLAUDE.md
Mesh source and formatPinned upstream by your geometry pipeline. Letting the assistant pick a different format mid-project breaks reproducibility.
Element family and polynomial orderOne number with outsized effect on convergence rates. Easy to set wrong, easy to change silently.
Error norms reported (L², H¹, energy, …)The norm you report defines what "convergence" means for your paper. A study that swaps norms between runs is not a study.
Figure conventions for convergence plotsLog-log axes, reference slope lines, file format, output directory. Three of these wrong and the figure is unusable in a paper.
Sanity-check valuesEvery PDE solver should report a small number (max of solution, energy, integral) you can eyeball against a known reference. Pin it once and forget.

CLAUDE.md is read at the start of every session. The assistant doesn't re-derive your conventions — it inherits them.

Exercise 1 15 min

Ask cold, let /init try, then add the rest

  1. Open cd exercises-pde/01-claude-md/. Read laplace_square.py — a deliberately under-specified Firedrake solve on a coarse UnitSquareMesh(8, 8) with hardcoded P1 and no convergence study.
  2. Phase 1 — no CLAUDE.md. Run claude on the host and ask add a convergence study for laplace_square.py. Note what Claude picks for norms, axes, file paths, and element order.
  3. Phase 2 — let /init try. /clear, run /init. It recovers the math in the docstring (manufactured solution, f = 2π²·u_exact, homogeneous Dirichlet BC), but has no way to know your conventions — figure format, log-log with reference slopes, the max(u_h) check, confirming element order.
  4. Phase 3 — add what /init couldn't know, then iterate. Merge those conventions into the CLAUDE.md, /clear, re-ask, compare. Add any missed rule and repeat.
Discussion: which conventions did Claude follow on the second try? Which did it miss? Add the missing rule to CLAUDE.md and ask once more — that's the whole loop.

Exercise 1 · debrief

What to look for in Claude's output

The improvement that respects a well-written CLAUDE.md should look something like this. Use it as a checklist, not an answer key.

Convention you wroteWhat it produces in the code
L² and H¹ errors with ratesA printed table with columns n, dofs, L2, L2 rate, H1, H1 rate. Rates computed as log2(prev/curr).
Log-log convergence plot with reference slopesTwo data lines (L², H¹) and two dotted reference lines at O(h^(p+1)) and O(h^p), anchored at the coarsest data point.
Figures saved, never shownNo plt.show(); output goes to figures/convergence.pdf.
Element order configurable + first-run confirmAn --order flag (default 1); on each --convergence run, prompt to confirm the order, persisting the choice in .element_order_confirmed.
Sanity check on every solveReport max(u_h) per level. The manufactured solution attains max = 1; values far from this signal trouble.

For P1 you should see L² rate → 2 and H¹ rate → 1 in the table, with the data lines parallel to the dotted reference lines on the plot.

Plan a fix for a broken convergence test

Here the deliverable is the plan.md, not code. Plan mode is the method; the plan you'd be comfortable approving is the artifact. Actually implementing the fix to watch the convergence rate recover is an optional next step.

PDE supplement · v1

Exercise 2 · framing

Why a plan first for a broken convergence test

The L-shape corner singularity is one of the most-taught problems in FEM, so the assistant has a strong textbook answer ready. Plan mode is what lets you check whether that textbook matches this code before any of it is written.

DecisionWhat plan mode forces you to settle in advance
Diagnostic chosenWhat "where the error is" actually means — element-wise |u_h − u_exact| vs. a residual indicator vs. eyeballing a plot. Skip this and you are refining without evidence.
Refinement mechanismWhether the implementation is one line or a hundred. MeshHierarchy + coordinate transformation, NetgenHierarchy with a marker, PETSc-level cell refinement — pick one before code is written.
Grading parametersWhat "graded toward the corner" actually means: a radius, a depth, an exponent, a marking criterion. Defaults pinned in the plan, not invented during implementation.
Out-of-scope disciplineKeeps the assistant from rewriting the rates harness while it is in there. Lose the harness and you lose the before/after comparison.
Visible before/afterThe same harness runs both times; that is the test. The plan has to leave the rates loop intact.

Exercise 2 · principle

Claude is a tool. You are the scientist.

On a classical pedagogical problem like this one, the assistant has seen the textbook answer many times and will draft it quickly. That cuts both ways: it tends to compress the diagnosis step — "we know it is the corner" — or to substitute a different refinement strategy than the one you asked for, because that is the variant it has seen more often.

Your job in plan mode is to decide how much structure to impose so the plan actually matches the problem in front of you. The "only the mesh is in scope" line in the prompt is one example of that structure. When you find yourself imposing the same structure session after session, promote it into your CLAUDE.md; that is what the file is for.

Claude Code becomes risky the moment you start trusting it to make major scientific decisions without checking. The pattern-match is fast and often plausible, and on a familiar problem it is very hard to tell a textbook reflex from real reasoning. The check has to come from you.

Exercise 2 15 min

Run the exercise

  1. cd exercises-pde/02-planning && claude; start the Firedrake container per ../01-claude-md/INSTALL.md.
  2. In the container, run python3 laplace_lshape.py. Note the L² rate stuck around 4/3 and the H¹ rate around 2/3 — far from the 2.0 and 1.0 the footer line says to expect.
  3. Press Shift+Tab twice to enter plan mode (look for the indicator at the bottom of the screen).
  4. Paste the prompt and submit. Do not approve yet. 
    The convergence in this code is suboptimal. Create a plan to
identify where the error is concentrated, then construct an a
priori mesh refinement strategy in the problem regions.

The polynomial order is fixed at P1; only the mesh is in scope.
  5. Read the plan against four highest-leverage questions: diagnostic mechanism? refinement primitive? grading parameters? what is out of scope? Push back on at least one.
  6. Save it without running the plan — approving would make Claude implement it. While still in plan mode, ask it to write the complete plan to plan.md and stop, then press Shift+Tab to leave plan mode without approving. That plan.md is the deliverable.
  7. Optional (if time permits): approve and let Claude implement, then re-run — watch how it paces (does it work through the plan's steps in order, or combine/skip them?); the L² rate should climb visibly toward 2.0.

Exercise 2 · debrief

What a plan worth approving contains

A plan that survives review on this exercise has six sections. Each one closes off a class of drift that would otherwise surface during implementation.

Plan sectionWhat it contains for this exercise
GoalOne paragraph. Modify laplace_lshape.py so the study refines more densely near the corner; target L² rate toward 2.0 and H¹ toward 1.0; keep the rates harness, manufactured solution, BC, and P1 untouched.
Step-by-stepInspect the mesh → implement the element-wise error diagnostic → implement the grading transformation → wire it into the rates loop → run and verify.
Out of scopeA posteriori adaptive marking. Dörfler loops. hp-refinement. Alternative element families. Touching the manufactured solution, the BC, or the polynomial order.

If you take the optional implement step, a successful run shows the rates moving visibly in the right direction — not necessarily reaching the textbook constants; closing the last fraction is FEM-tuning craft, out of scope here.

Exercise 2 · closing

What plan mode is really for

Plan mode produces text. The value is not the text — it is the questions the act of planning forces you to answer in advance. They are the same questions you would have hit during implementation, just in an order where you can still afford to redirect cheaply. On a problem the assistant knows well, those questions are also the ones the assistant most wants to skip; on a problem it has never seen, they are the ones you most need to spell out. Either way, the scientist is you.

Take the conventions you found yourself imposing back to your CLAUDE.md. The next exercise builds on that file again.

Exercise 3 · skills

Verify a CFL condition with a skill

Skills are the workshop's mechanism for packaging the check your team runs every project. The check is the pretext; the skill is the artifact.

PDE supplement · v1

Exercise 3 · framing

Why a skill (and not just a function)?

A skill is a folder Claude loads on demand — SKILL.md plus helper scripts. The shape is simple; what makes it earn its keep is four properties that a private function or a CLAUDE.md note do not have.

PropertyWhy it earns the skill its keep
Triggered by description, not importClaude routes a natural-language prompt to the skill via the description: field — you do not tell it which file to run.
Stateless and deterministicSame input gives the same output. No external service to be down, no license dance.
Versioned in the repoThe check travels with the project. git clone brings the team's CFL check along, no per-developer setup.
The description is the contractExtend the helper and the description must change too. An extension that does not change the description is invisible to future natural-language requests.

Exercise 3 15 min

Run the exercise

  1. cd exercises-pde/03-skills
  2. Install the skills into the project-local hidden directory: 
    mkdir -p .claude
cp -R skills .claude/
  3. Launch claude. No interactive container needed — the skill calls docker run --rm itself when it fires.
  4. Paste the prompt — note that it does not name the skill; description-routing is what is being tested: 
    Verify that the proposed timestep in candidate.json is CFL-safe
for the wave problem defined in problem.py.
  5. Approve the docker command when Claude prompts you (the default permission policy flags new shell commands). Read the PASS output — note the safe bound and the per-layer summary.
  6. Edit candidate.json to {"dt": 0.05}. Re-ask. Confirm FAIL and that the violators localize to the bottom (high-c) layer.
  7. Restore dt below safe. Re-run. Back to PASS.

Exercise 3 · debrief

What you should see

Read Claude's behavior as much as the skill's output. The pass / fail / restore loop is deterministic; what varies between learners is how cleanly the skill mechanism worked end-to-end.

What you should seeDetail
Passing runPASS: dt = 0.002, safe ~ 8e-3. Limiting cell is in the bottom layer; per-layer summary shows the bottom roughly 2× tighter than the top.
Failing runFAIL: dt = 0.05. Several hundred violators, all in the bottom layer (between the two per-layer minima).
What flippedThe lead (PASS → FAIL). The safe bound itself did not change — it is a property of the discretization, not of dt.
Behavior to watchDid Claude run the skill, or recompute the check inline? Did it use the docker-wrapped command, or try bare python3 on the host? Did it show the full output, or paraphrase?

Exercise 3 · closing

Skills as your team's checklists

Every research group has a small set of checks they re-derive every project — KKT residuals, CFL bounds, convergence-rate fits, mesh-quality reports, citation-format validators. Skills are the place those checks earn back their authorship cost. The price is small (a SKILL.md and a script); the payoff is that everyone in the group, and every future Claude session, runs the same check the same way.

If you only remember one rule about skills: the description is the contract. When you extend a skill, update its description in the same commit — otherwise the new capability is invisible to natural-language routing, and the next person to ask for it will silently re-derive what you already built.

Exercise 4 · LOGBOOK.md

Bootstrap a LOGBOOK.md from lab notebooks

LOGBOOK.md is how this session leaves something behind for the next one. The synthesis is the pretext; the file is the artifact.

PDE supplement · v1

Exercise 4 · framing

Why a LOGBOOK.md (and not just more CLAUDE.md)?

Both are markdown files at the project root. Both get read by Claude on session start. They answer different questions, and confusing the two is the most common decay path for both.

QuestionAnswer
What does CLAUDE.md answer?How does this project work. Stack, file layout, conventions, hard rules.
What does LOGBOOK.md answer?What have we learned. Decisions, parameters, dead ends, open questions.
Growth patternCLAUDE.md changes rarely. LOGBOOK.md grows monotonically as experiments accumulate.
The testIf knowing a fact depends on having run experiments, it belongs in LOGBOOK.md. Otherwise CLAUDE.md.

Exercise 4 10 min

Run the exercise

  1. cd exercises-pde/04-logbook — three dated lab-notebook entries are already there. No container needed; this is a writing exercise.
  2. Launch claude and paste: 
    Read everything under notebooks/ and produce LOGBOOK.md with the
sections: Decisions, Parameters, Dead Ends, Open Questions. Each
entry should cite the notebook file it came from.
  3. Trim aggressively. Drop anything a future session would not consult. If the file is longer than the source notebooks combined, push back — that is copying, not synthesis.
  4. With the trimmed file in place, paste: 
    Given LOGBOOK.md, what would be the most informative next experiment
to run? Justify in two sentences.
    A good answer cites a specific dated entry; a hand-wave is a teachable failure.
  5. Close with the two-minute ritual: 
    Summarize what we did in this session, append as a dated entry to
LOGBOOK.md, and overwrite STATUS.md with where we are now.
LOGBOOK.md isn't limited to .md — Claude reads source, LaTeX, and data too. The optimization track's Exercise 04 distills a whole multi-format project (plans/ + src/ + tex/) into one logbook file — worth a look.

Exercise 4 · debrief

What a good synthesis looks like

The exercise tests editorial judgment more than computational behavior. Read the synthesized file and Claude's next-experiment answer against this checklist.

What a good synthesis doesWhat to push back on
Cites the source notebook by filename on every entry."See notebooks" or no citations at all.
Comes in well under the combined length of the three source notebooks.Longer than the inputs — that is re-typing, not synthesis.
Dead Ends section preserves the why a path was abandoned.Verdict only ("abandoned X"). The reason is what stops the re-attempt six months later.
Next-experiment answer names a specific dated entry and proposes a one-session scope.Generic "more research needed," or a multi-week project that ignores what LOGBOOK.md said.
Surfaces any conflict between two notebooks rather than smoothing it over.Silent merge of two entries that disagree — destroys the most valuable information.

Exercise 4 · closing

LOGBOOK.md as next-session inheritance

Every session is downstream of a hundred forgotten ones — your own from last month, your colleagues', the one you'll resume on a different machine in three weeks. LOGBOOK.md is the file that decides what the next session knows. Without it, every resumption pays the same "what was I working on" tax.

The two-minute end-of-session ritual is the workshop's most lopsided ROI move: append a dated entry to LOGBOOK.md, overwrite STATUS.md with where we are now. Two minutes now; saves the first ten minutes of the next session, every time.
