Exercise 1 · framing

Why a linear algebra researcher needs CLAUDE.md

"Find the spectrum" is under-specified in three ways at once. A CLAUDE.md turns each silent guess into an explicit, up-front question.

DecisionWhy it belongs in CLAUDE.md
Matrix type (symmetric? Hermitian? nonsymmetric?)Decides eigvalsh vs eigvals, and whether the eigenvalues are even real.
Storage (dense / sparse / matrix-free)Decides the routine. Densifying a large sparse matrix to reuse a dense solver is the classic blunder.
Eigenvalues vs singular valuesThey coincide only for symmetric PSD matrices; "spectrum" silently hides which one you meant.
Full spectrum vs dominant-kAll 5000 eigenvalues is rarely the goal; iterative methods give the top few cheaply.
Sanity valuesnnz / density / condition number — eyeball-able numbers that catch a wrong-type result.

Exercise 1 15 min

Pin type, storage, and meaning

  1. cd exercises-lin_alg/01-claude-md. Two matrices ship — matrix_A.npy and matrix_B.npz. One is dense, one is sparse; one is symmetric, one is not. Nothing tells you which.
  2. Phase 1, no CLAUDE.md. Ask: find the spectrum of these matrices. Watch: does Claude identify type + storage first? does it densify the sparse one? does "spectrum" become eigenvalues or singular values — and does it ask, or just guess?
  3. Phase 2. Write a CLAUDE.md that forces "identify type and storage, then ask eigenvalues-vs-singular-values and full-vs-dominant-k before computing." /clear and re-ask the same prompt.
  4. Compare. Claude should now report that A is dense nonsymmetric and B is large sparse symmetric, then ask what you want before touching the data.
Discussion: which of the three ambiguities can Claude resolve on its own by inspecting the files, and which does it need you to pin in the CLAUDE.md?

What the CLAUDE.md changed

The payoff is the contrast between the cold run and the run with conventions in place. Use it as a checklist, not an answer key.

SignalPhase 1 (no CLAUDE.md) → Phase 2 (with it)

Plan an SVD of a linear operator

Exercise 2 · framing

Why plan an SVD you can't form

heat_operator.py is one implicit heat step, available only as apply(v). Planning its SVD forces choices that a "just run svds" reflex skips.

DecisionWhat plan mode forces you to settle in advance
Matrix-free realityThe operator has no dense form. The plan must wrap apply as a LinearOperator, not call np.linalg.svd.
How many modesDominant-k, not the full SVD of a 1600-dim implicit operator.
Algorithm & costsvds / randomized SVD — each matvec is one sparse solve, so cost is measured in applies.
ParametersN, dt define the operator; unstated, results aren't comparable.
VerificationA small-N dense cross-check is the cheap way to trust a matrix-free result.
Linear algebra supplement · v1

Exercise 2 · principle

Claude is a tool. You are the scientist.

The operator behaves like a matrix — you can hit any vector with it — but you cannot index it, transpose it, or form it densely; each apply is a sparse solve. The assistant will paper over that if you let it: the common failure is a plan that quietly calls np.linalg.svd on a matrix that doesn't exist, or densifies the operator to get one.

Your job in plan mode is to force the matrix-free reality — and the choices it implies (how many modes, which algorithm, what parameters) — into the plan up front.

A plan that densifies a matrix-free operator looks perfectly reasonable until you notice the matrix was never supposed to exist. The check has to come from you.

Exercise 2 15 min

Run the exercise

  1. cd exercises-lin_alg/02-planning. Run it for 10 time steps to see the operator drive the physics: 
    python heat_operator.py --steps 10
    A central hot spot diffuses; the peak temperature decays step by step.
  2. Press Shift+Tab twice to enter plan mode.
  3. Paste the prompt and submit — do not approve yet. It carries a matrix-free scope clause: 
    Plan a singular value decomposition of the linear operator in
heat_operator.py. The operator is matrix-free: it exposes only
apply(v). Do not form or densify the matrix.
  4. Read the plan against four questions: matrix-free? dominant-k? algorithm + cost? parameters pinned? Push back on at least one.
  5. Save it to plan.md — that file is the deliverable.

Exercise 2 · closing

What plan mode is really for

Plan mode produces text. The value isn't the text — it's the questions the act of planning forces you to answer while redirection is still cheap. On a matrix-free operator, the first of those questions is the one the assistant most wants to skip: you cannot form this matrix, so how will you decompose it? Approve the plan only once it answers that on your terms.

Take the conventions you found yourself imposing back to your CLAUDE.md — the next exercise builds on that file again.

Exercise 3 · framing

Why a skill (and not just a function)?

Two skills — qr-iteration (dense) and lanczos (sparse or matrix-free) — split "find the singular values" by storage. Four properties earn them their keep over a private function.

PropertyWhy it earns the skill its keep
Triggered by description, not importClaude routes a plain request to the skill via its description: — you don't name the file.
Stateless and deterministicSame object, same singular values. No external service, no license dance.
Versioned in the repogit clone brings the team's QR and Lanczos routines along, no per-developer setup.
The description is the contractExtend the helper and the description must change too, or the new capability is invisible to routing.

Run the exercise

  1. cd exercises-lin_alg/03-skills. Install the skills: 
    mkdir -p .claude
cp -R skills .claude/
    problem.py exposes the three objects from Exercises 01–02.
  2. Ask — without naming a skill (routing is what's tested):

    Claude should route qr-iteration to the dense matrix and lanczos to the sparse matrix and the operator.

Exercise 3 · closing

Skills as your team's checklists

Every group re-derives a small set of computations each project — QR and Lanczos SVDs, condition-number probes, residual fits. Skills are where those earn back their authorship cost: the price is a SKILL.md and a script; the payoff is that everyone, and every future Claude session, runs the same method the same way.

If you remember one rule about skills: the description is the contract. Extend a skill and update its description in the same commit — otherwise the new capability is invisible to natural-language routing.

Exercise 4 · LOGBOOK.md

Bootstrap a LOGBOOK.md from lab notebooks

LOGBOOK.md is how this session leaves something behind for the next one. The synthesis is the pretext; the file is the artifact.

Linear algebra supplement · v1

Exercise 4 · framing

Why a LOGBOOK.md (and not just more CLAUDE.md)?

Both are markdown files at the project root, both read on session start. They answer different questions, and confusing the two is the most common decay path for both.

QuestionAnswer
What does CLAUDE.md answer?How does this project work. Stack, conventions, hard rules.
What does LOGBOOK.md answer?What have we learned. Decisions, parameters (rank 8, cadence 10), dead ends, open questions.
Growth patternCLAUDE.md changes rarely; LOGBOOK.md grows as experiments accumulate.
The testIf knowing a fact depends on having run experiments, it belongs in LOGBOOK.md. Otherwise CLAUDE.md.

Exercise 4 10 min

Run the exercise

  1. cd exercises-lin_alg/04-memory — three dated notebooks (a heat-equation testbench with an SVD preconditioner re-initialized every 10 steps). A writing exercise; no environment needed.
  2. Paste: 
    Read everything under notebooks/ and produce LOGBOOK.md with the
sections Decisions, Parameters, Dead Ends, Open Questions — each
entry citing the notebook it came from.
  3. Trim aggressively. Drop anything a future session wouldn't consult; if it's longer than the notebooks combined, that's copying, not synthesis.
  4. Use it to pick a next experiment — Given LOGBOOK.md, what's the most informative next experiment? Justify in two sentences. A good answer cites a specific entry.
  5. Close with the two-minute ritual: append a dated entry to LOGBOOK.md and overwrite STATUS.md.

Exercise 4 · debrief

What a good synthesis looks like

The exercise tests editorial judgment more than computation. Read the synthesized file and the next-experiment answer against this checklist.

What a good synthesis doesWhat to push back on
Cites the source notebook on every entry."See notebooks" or no citations at all.
Comes in well under the combined length of the notebooks.Longer than the inputs — re-typing, not synthesis.
Dead Ends keep the reason a path was abandoned.Verdict only ("abandoned per-step re-init") — the 2.4×-slower reason is what stops the re-attempt.
Next-experiment answer names a specific entry.Generic "study convergence more."
Surfaces the cadence trade-off (10 works; 1 and 40 fail).Smoothing it to "re-init periodically" — that destroys the most valuable finding.

Exercise 4 · closing

LOGBOOK.md as next-session inheritance

Every session is downstream of a hundred forgotten ones — your own from last month, your colleagues', the one you'll resume on a different machine in three weeks. LOGBOOK.md is the file that decides what the next session knows. Without it, every resumption pays the same "what was I working on" tax.

The two-minute end-of-session ritual is the workshop's most lopsided ROI move: append a dated entry to LOGBOOK.md, overwrite STATUS.md with where we are now. Two minutes now; saves the first ten minutes of the next session, every time.
