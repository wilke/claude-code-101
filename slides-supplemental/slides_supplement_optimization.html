Exercise 1 · framing

Why an optimization researcher needs CLAUDE.md

Optimization research is full of recurring micro-decisions — small, stable choices that shape every result and that you don't want to re-justify in every prompt. CLAUDE.md is where those live.

DecisionWhy it belongs in CLAUDE.md
Variable naming (primal, dual, multipliers)Pinned by the math. Letting the assistant pick its own letters mid-project breaks readability across files and notes.
Solver choice and modeling stackPyomo vs CasADi vs raw scipy.optimize. IPOPT vs BARON vs HiGHS. Each comes with assumptions about problem shape; switching silently is silently wrong.
Tolerances and terminationOne number with outsized effect on what counts as "converged". Easy to set wrong, easy to change without noticing.
Figure conventions for convergence plotsSemilog axes, ‖∇f‖ vs iteration, file format, output directory. Three of these wrong and the figure is unusable in a paper.
Sanity-check valuesFinal gradient norm, function value at optimum, KKT residual. Eyeball-able numbers that distinguish "solved" from "lucky".

CLAUDE.md is read at the start of every session. The assistant doesn't re-derive your conventions — it inherits them.

Exercise 1 10 min

Write your own CLAUDE.md

  1. cd exercises/01-claude-md/
  2. Read the included rosenbrock.py — a deliberately under-tested SciPy solve.
  3. Run claude and ask: /init. Read the generated CLAUDE.md.
  4. Edit it. Add: project goal, your preferred tolerance, and one "don't" rule.
  5. Then ask Claude: add a convergence plot for rosenbrock.py using our conventions.
Discussion: did Claude follow your conventions? What did it miss? Add the missing rule to CLAUDE.md and try again — that's the whole loop.

Exercise 1 · debrief

What to look for in Claude's output

The improvement that respects a well-written CLAUDE.md should look something like this. Use it as a checklist, not an answer key.

Convention you wroteWhat it produces in the code
Tolerance pinned (e.g., 1e-8)Solver call passes tol=1e-8 rather than defaulting silently. Iteration count and final gradient norm reported in the script's output.
Convergence plot conventionsSemilog y-axis on ‖∇f‖, linear x-axis on iteration count, with axis labels matching what your CLAUDE.md specified.
Figures saved, never shownNo plt.show(); output goes to figures/convergence.pdf (or whatever path you pinned).
Gradient history capturedA callback passed to minimize that records ‖∇f‖ at each iteration, then plotted — not a hand-rolled second solve.
"Don't" rule respectedWhatever you forbade (e.g., no extra imports, no scipy.optimize.minimize for constrained problems) actually doesn't appear in the diff.

If Claude missed a convention on the first try, add a sharper rule and ask again — that's the whole loop.

Exercise 3 · framing

Why a skill (and not just a function)?

A skill is a folder Claude loads on demand — SKILL.md plus helper scripts. The shape is simple; what makes it earn its keep is four properties that a private function or a CLAUDE.md note do not have.

PropertyWhy it earns the skill its keep
Triggered by description, not importClaude routes a natural-language prompt to the skill via the description: field — you do not tell it which file to run.
Stateless and deterministicSame input gives the same output. No external service to be down, no license dance — unlike an MCP.
Versioned in the repoThe check travels with the project. git clone brings the team's KKT check along, no per-developer setup.
The description is the contractExtend the helper and the description must change too. An extension that does not change the description is invisible to future natural-language requests.

Exercise 3 15 min

Run the exercise

  1. cd exercises/03-skills
  2. Install the skills into the project-local hidden directory: 
    mkdir -p .claude
cp -R skills .claude/
  3. Launch claude. The helper is pure Python + NumPy — no container needed for this exercise.
  4. Paste the prompt — note that it does not name the skill; description-routing is what is being tested: 
    Verify the candidate solution in solution.json against the QP
defined in problem.py.
  5. Approve the shell call when Claude prompts you (default permission policy flags new commands). Read the PASS output — all five KKT residuals near machine precision.
  6. Edit solution.json: change z[0] from 0.0 to 0.5. Re-ask. Confirm FAIL with non-zero stationarity and complementarity residuals.
  7. Restore. Re-run. Back to PASS.

Exercise 3 · debrief

What you should see

Read Claude's behavior as much as the skill's output. The pass / fail / restore loop is deterministic; what varies between learners is how cleanly the skill mechanism worked end-to-end.

What you should seeDetail
Passing runAll five residuals at or near machine precision: stationarity ~ 1e-17, eq feas = 0, min x_i > 0, min z_i >= 0, complementarity = 0. Verdict PASS.
Failing runAfter corrupting z[0]: stationarity jumps to ~5e-1, complementarity to ~7e-2. The two residuals that flip localize the defect to the dual block.
What flippedThe lead (PASS → FAIL) and exactly the residuals that depend on z. The other three are unchanged — that is the whole point of the per-block layout.
Behavior to watchDid Claude run the skill, or recompute the KKT residuals inline? Did it show the full output, or paraphrase? Did it second-guess the verdict?

Exercise 3 · closing

Skills as your team's checklists

Every research group has a small set of checks they re-derive every project — KKT residuals, CFL bounds, convergence-rate fits, mesh-quality reports, citation-format validators. Skills are the place those checks earn back their authorship cost. The price is small (a SKILL.md and a script); the payoff is that everyone in the group, and every future Claude session, runs the same check the same way.

If you only remember one rule about skills: the description is the contract. When you extend a skill, update its description in the same commit — otherwise the new capability is invisible to natural-language routing, and the next person to ask for it will silently re-derive what you already built.

Exercise 5 · framing

MCP vs skill vs CLAUDE.md note

Three places to put a capability. They overlap, but each one earns its keep in a different regime.

MechanismWhen it earns its keep
CLAUDE.md noteOne-shot conventions ("always normalize Q before solve_qp"). No code needed; Claude reads CLAUDE.md every session.
SkillStateless, reusable inside this project, ships in the repo. The KKT-residual check from Exercise 3 is the canonical shape.
MCPPersistent state worth keeping warm across calls; or it talks to something Claude can't run inline (a licensed solver, a remote cluster); or it needs to be available to many sessions across many projects.
The toy in this exercise is honestly skill-shaped — the MCP shell is what's being taught. The stretch (swap cvxpy for a warmed-up PETSc/TAO process) is where the persistent-state justification actually bites.

Exercise 5 15 min

Run the exercise

  1. cd exercises/05-mcp/ and pip install -r requirements.txt.
  2. Open toy_solver_mcp.py — a ~60-line MCP server exposing solve_qp(Q, c, A, b) via cvxpy.
  3. Register the server: 
    claude mcp add toy-solver --command python --args toy_solver_mcp.py

    ⚠ Verify before running live: recent claude versions removed the --command/--args flags in favor of claude mcp add toy-solver -- python3 toy_solver_mcp.py. Run claude mcp add --help to confirm the syntax on your install. On Homebrew macOS, only python3 is on PATH (no python).

  4. Launch claude in this directory and paste: 
    Solve the QP in problem.json using the toy-solver MCP and report
the structured result.
  5. Edit problem.json (change c, or perturb a row of A) and re-prompt. Confirm Claude re-calls the MCP rather than recycling the prior answer.

Exercise 5 · debrief

What "MCP routing" looks like

The artifact to inspect is not the optimum — it is the call. Read what Claude did against this checklist.

Good routingWhat to push back on
Calls toy-solver:solve_qp by name.Solves the QP inline with cvxpy — bypasses the MCP entirely.
Arguments at the wire match problem.json verbatim — Q, c, A, b all in the right shape.Reshaped Q, rounded entries, or symmetrized inputs in passing without telling you.
Reports the structured payload (status, optimum, primal, dual) verbatim alongside any prose.Paraphrases as "optimal, value ≈ -3.2, looks fine" and drops the JSON.
Re-calls the MCP when problem.json changes.Recycles the prior answer — treats the first call as a memo, not a tool.
Treats the MCP as the source of truth.Cross-checks with an inline cvxpy solve, then second-guesses the MCP.

Exercise 5 · closing

MCP = capability that survives the session

The artifact this exercise produced is not the QP solver. It is the registered tool name — toy-solver:solve_qp — that any future session in this folder can invoke without setup, without re-explaining, without re-implementing.

The stretch is where the design compounds: swap cvxpy for PETSc/TAO with a warmed-up solver process and the MCP starts amortizing setup cost across every call. That is the regime where "MCP, not skill" stops being a teaching point and starts being the design that actually wins. The capstone (Exercise 6) is where multiple MCPs meet in a single workflow.
